Edit 1 — add a new subsection immediately after ### Data field:

HTTP status conventions
These rules fix the HTTP status code a route returns, independent of the envelope's found flag.

Success and honest absence both return 200. A found: true response and a found: false response are both the route working correctly. found: false is honest absence, not an error, and must not be signaled with 404. 404 is reserved for a route or path that does not exist. Every envelope-carrying route returns 200 whenever it produced a well-formed envelope, regardless of the found value.
Invalid client input returns 400 with a bare error object — no envelope. When a request is rejected before any source lookup is attempted — a missing required parameter, an out-of-range or unrecognized enum value, a malformed key — the route returns HTTP 400 with the body { "error": "<machine_code>", "message": "<human_readable>" } and nothing else. It does NOT carry the response envelope: no source was queried, so there is no provenance to record, and per the governing rule the FERNS-produced error text must never be placed inside data. The error value is a stable machine-readable code (e.g. invalid_input); message is human-readable guidance.
