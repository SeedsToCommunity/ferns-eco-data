# Ann Arbor Native Plant Nursery — Data Capture History

This document describes how the Ann Arbor Native Plant Nursery (AA NPN) dataset held by FERNS was originally captured from `nativeplant.com`. It is the surviving record of that work. The capture code has been removed from the codebase; what was learned from live-site exploration is preserved here, in prose, rather than in implementation. The dataset itself lives as static data files inside the `ann-arbor-npn` Internal Data Provider package, and the upstream site is treated as defunct.

## What was scraped

The upstream source was Greg Vaclavek's website at `http://nativeplant.com`. The site catalogued approximately 130 native plant species available from his Ann Arbor-area nursery, with botanical descriptions, growing conditions, photographs, and pricing.

The capture pulled the following fields for each species:

- **Acronym** — Greg's three-to-six-letter species code, used as the canonical key (e.g. `GENAND` for Closed Gentian).
- **Latin name** — the accepted scientific name.
- **Latin synonym (Greg's nomenclature)** — an alternate latin name Greg used, often older taxonomy. Captured as an `a.k.a.` value from the species page header.
- **Common name** — one or more common names, separated by `;` or `,`.
- **Light requirement** — sun/shade descriptor.
- **Moisture requirement** — moisture descriptor.
- **Height** — typical mature height range.
- **Bloom time / flowering time** — descriptive timing.
- **Flower color**.
- **Physiography** — plant type (Forb, Grass, Sedge, Shrub, Tree, Vine, Fern).
- **Numeric scores** — Greg's internal `S` (light), `W` (moisture), and `C` (companion/color) scores.
- **Notes** — Greg's free-text notes about the species.
- **Michigan range** — regions of Michigan where the species is native (SE, SW, NL, UP).
- **Price/size information** — sizes and prices Greg offered.
- **Photographs** — full-resolution JPEGs from the species detail page, with captions and a kind tag inferred from caption text (`photograph` versus `drawing` for pen-and-ink illustrations).

## How the capture worked

### Two-endpoint strategy

The site exposed two relevant endpoints:

1. **`GET /plants/search/report`** — a Zope/MySQL report endpoint. With the right combination of `show_*` query flags, it returned all ~130 species in a single HTML response with structured columns for every textual field listed above. This was the primary data source.

2. **`GET /plants/plant_page_template?Acronym={code}`** — the individual species detail page. This was used only for two things: the full-resolution image URLs and Greg's `a.k.a.` latin synonym. All other textual data came from the report endpoint, not from the detail pages.

This split was a deliberate efficiency choice. An earlier exploration suggested needing two requests per species (a list page plus a detail page, about 260 total requests). The discovery that the report endpoint returned everything textual in one request reduced the total to roughly 131 requests (one report plus one per species for images).

### Report parsing

The report endpoint returned HTML, not structured data. The parser worked by:

1. Splitting the HTML on `<tr>` boundaries.
2. Filtering to rows containing a `plant_page_template?Acronym=XXX` link (this excluded header rows, separator rows, and anything without a species).
3. Extracting each row's `<td>` contents in order.
4. Stripping HTML tags and decoding common entities (`&nbsp;`, `&amp;`, `&lt;`, `&gt;`) to get plain text.
5. Mapping column positions to fields. The column order was discovered by inspection of the live site with all `show_*` flags enabled and is documented in the source.
6. Normalizing empty strings and `"n/a"` values to `null`.

The required minimum was 15 columns per row; the price column was sometimes absent and was tolerated.

### Detail page parsing

For each species, the detail page was fetched to extract:

- **The `a.k.a.` latin synonym** — extracted from an `<h3>(a.k.a. Synonym Name)</h3>` pattern in the page header. One species, `GENAND` (Closed Gentian), had a common name in the synonym slot rather than a latin name; this was indexed anyway so that common-name lookups for that species would still resolve correctly.
- **Image blocks** — full-resolution image URLs were extracted from `<a href="/plant_images/ACRONYM/N.jpg">` anchors. Captions, when present, were extracted from a `<small>` tag that followed the anchor. Duplicate URLs within the same page were deduplicated. Position numbering was sequential within each species.

### Image handling

Image handling was the most defensive part of the capture, because the upstream site had a mix of valid JPEGs, broken images, oversized HTML error pages returned with image content-types, and images in non-JPEG formats. The handling rules were:

1. **Fetch with a 20-second timeout** and a polite `User-Agent` identifying FERNS.
2. **Reject anything under 512 bytes** — too small to be a real photograph.
3. **Verify JPEG magic bytes** (`0xFF 0xD8 0xFF`) on the first three bytes of the response body. Anything else was rejected even if the content-type header claimed JPEG.
4. **Upload validated bytes to Cloudinary** at a public ID of the form `ann-arbor-npn/{acronym-lower}/{position}`, with the caption attached as a Cloudinary `context` tag.
5. **Omit anything that failed validation or upload.** No fallback to the original source URL was kept — the design choice was to store only verifiable, persistent image references, not ephemeral upstream links that might rot.

Image kind was inferred from the caption: captions containing "pen & ink" or "drawing" were tagged as `drawing`; everything else was tagged as `photograph`.

### Alias indexing

Each species record was associated with a set of lookup aliases so that queries by acronym, latin name, latin synonym, or common name would all resolve to the same record. Aliases were built as a deduplicated set of:

- The acronym (lower-cased).
- The latin name (lower-cased).
- The latin synonym, if present, indexed as-is (the GENAND exception above accepted).
- Each common name segment (split on `;` and `,`, trimmed, lower-cased).

These aliases were stored in a separate table (`npn_name_aliases`) keyed by the lower-cased alias string, pointing back to the canonical acronym.

### Politeness

Between detail-page requests, the capture slept for a randomized 300–500ms (a 300ms minimum plus a 0–200ms jitter). This kept the request rate at roughly two per second, which was deliberate restraint against a small one-person site.

### Trigger and idempotency

The capture was triggered either by a `POST /api/ann-arbor-npn/import` admin endpoint or by an `autoImportNpnIfEmpty` startup check. The latter would only run if the species table was empty, which prevented accidental re-imports on every server restart. Both species records and alias records used upserts (`onConflictDoUpdate`), so re-running the capture would refresh records in place rather than duplicating them.

### Filtering

The import function accepted an optional list of acronyms. If provided, only those species were captured. This was used during development to test the pipeline against a small subset before running the full ~130-species capture.

## What was captured

The result of the capture was:

- A `npn_species` table with one row per species, containing all textual fields plus an `images` array (each entry holding a Cloudinary URL, caption, position, and kind) and a `source_url` pointing back to the original detail page.
- A `npn_name_aliases` table mapping every lookup alias to its canonical acronym.
- A set of validated JPEG images hosted at Cloudinary, keyed by acronym and position.
- A `scraped_at` timestamp on each species record, set to the time the record was inserted or last refreshed.

## Migration to static data files

The PostgreSQL tables described above were the working storage for the captured data, but they are no longer part of the FERNS architecture. The upstream site is defunct — even though `nativeplant.com` may still respond, no further captures will ever run against it. FERNS now treats the dataset as a captured snapshot rather than a cached scrape.

The data was exported from `npn_species` and `npn_name_aliases` into static data files inside the `ann-arbor-npn` Internal Data Provider package at `lib/internal-data-providers/src/ann-arbor-npn/`. The Source Interface loads these files at startup and serves all queries from memory. The PostgreSQL tables, the import code, the Cloudinary credentials, and the `POST /api/ann-arbor-npn/import` route have all been removed from the live system.

The Cloudinary-hosted images themselves remain in place and are referenced by their persistent URLs in the static data; that part of the original design — favoring persistent hosted URLs over ephemeral upstream links — proved correct and survives the migration unchanged.

This document is therefore the record of what was figured out. The two-endpoint strategy, the report column ordering, the magic-byte JPEG validation, the GENAND alias handling, the alias-building rules — these decisions came from live-site exploration that can no longer be repeated, and they are preserved here because the implementing code is not. If a similar capture is ever needed for another defunct source, this account is the reference.
