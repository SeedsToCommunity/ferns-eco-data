Good analysis. Here's how to handle each open point.
Habitat field — include it in the static type and data even if sparsely populated. The Source contains the field; the data should reflect that. The Adapter's current routes may not surface it today, but that's an Adapter decision, not a reason to drop it from the Source. A future Adapter route might use it. Separation of concerns: the Source returns everything it has; what the Adapter does with each field is a separate decision.
Per-species source_url — include it as a field on the species record returned by the Source Interface. Applications may call these URLs to link to species detail pages on nativeplant.com. They work today; the upstream is defunct as a business but the site still responds. The Adapter passes the field through unchanged; the application decides what to do with it (link to it, hide it, mark it as "external reference," verify it, ignore it — all application concerns). The Adapter does not resolve, validate, or transform the URL.
scraped_at — replace per-record timestamps with a single module-level constant ANN_ARBOR_NPN_SNAPSHOT_DATE set to the actual capture date. Per-record scraped_at made sense when records could be re-captured at different times; with a static snapshot, one date covers everything.
Alias index — rebuild at module load from the in-record acronym, latin name, latin synonym, and common name fields using the same logic as the original buildAliases(). This is pure string derivation from in-record data with no external dependency, so it's safe to derive rather than ship as a parallel constant.
Cloudinary image URLs — these are not derivable. They are captured artifacts of what succeeded at upload time, and what's in the database now is the only source of truth. Read them straight out of the DB during the data extraction step and copy verbatim into the static data file. Do not try to rebuild URLs from public_id patterns — the URLs in the data reflect actual upload outcomes (some images failed JPEG validation, some failed upload, some species have gaps in their position sequence), and that information exists only in the captured data.
The extraction step generally — treat it as a read-and-serialize pass over the DB rows. Any field that came from the capture (image URLs, captions, positions, kinds, the species text fields, source URLs) gets copied as-is. Derived structures (the alias index) are derived only when they can be derived from in-record data with no external dependency.
technical_details should state explicitly:

The dataset is a captured snapshot as of ANN_ARBOR_NPN_SNAPSHOT_DATE. The upstream company is defunct.
Per-species source_url values point at nativeplant.com detail pages. These URLs were valid at capture time and may still resolve, but the upstream is unsupported and they may stop working without notice. FERNS does not verify them.
Species image URLs point at Cloudinary. The images are not in the repository; they live at a third-party CDN controlled by FERNS. The snapshot is fully self-contained for textual data; image availability depends on the Cloudinary account remaining live.
The 7a clause: this Source is a captured snapshot, not a live or refreshable Source. No further captures will run.

What gets removed — import.ts in full, the POST /api/ann-arbor-npn/import route, the autoImportNpnIfEmpty startup call, all db/npnSpeciesTable/npnNameAliasesTable imports from the live code path. The capture code does not need to be preserved in the codebase; a separate document records the capture process for historical reference.
DB tables — drop npn_species and npn_name_aliases in a separate follow-up task after the static version is verified working. Don't conflate "switch to static data" with "delete the tables that proved we can switch."
Identifier renames — same pattern as the previous three IDPs: NPN_ → ANN_ARBOR_NPN_, services directory npn/ → ann-arbor-npn/. Route paths already match the source_id, so no HTTP path changes are needed; only the file/directory renames and identifier renames apply.
