# Michigan Flora Connector Guide

**Source**: Michigan Flora Online  
**Website**: https://michiganflora.net  
**Maintainer**: University of Michigan Herbarium  
**Data Type**: Michigan vascular plant records — ecological metrics, descriptions, county occurrences, synonyms  
**Authentication**: None required for either interface  
**Last Verified**: December 2025

---

## Overview: Two Separate Interfaces

Michigan Flora exposes its data through two fundamentally different interfaces, and this connector uses both. They are not redundant — each provides distinct fields. They must be understood separately.

| Interface | What it provides | Access method |
|-----------|-----------------|---------------|
| **CSV dataset** | C value, wetness (W), physiognomy, duration, native status (text), common name, family, acronym | Local file, loaded into memory at startup |
| **REST API** | plant_id, native status (coded), description text, county occurrences, synonyms, record URL, state status, images | HTTP calls to `https://michiganflora.net/api/v1.0/` |

**Critical**: The Michigan Flora **website** (`michiganflora.net`) is a Vue.js single-page application. Making an HTTP GET request to a species page URL returns only the SPA shell HTML — no species data, no rendered content. Do not attempt to scrape the website directly. All structured data must come from either the local CSV or the REST API.

## Interface 1: The CSV Dataset

### File

```
cache/MichiganFlora/MichiganFloraSpeciesDatabase_Michigan_2024.csv
```

The CSV file is the canonical offline dataset. It is not generated by the pipeline — it is a downloaded artifact from Michigan Flora, versioned by year. The filename encodes the year: `2024`. If a newer annual release is obtained, it would be a new file with a new year suffix, not a replacement of this one.

### Dataset Statistics (from the file's own metadata rows)

- Total species: 2,873
- Native species: 1,810
- Non-native species: 1,063
- Total mean C: 4.1
- Native mean C: 6.5

### CSV Structure — The Header Is Not on Line 1

**This is the single most important parsing detail**: The CSV file contains metadata rows above the actual data header. The first ~11 lines look like this:

```
Michigan
2024
Merjent
"Michigan (MI) , 2024"

"Total Species:",2873
"Native Species:",1810
"Non-native Species:",1063
"Total Mean C:",4.1
"Native Mean C:",6.5

```

Line 12 is the actual column header:

```
"Scientific Name",Family,Acronym,Native?,C,W,Physiognomy,Duration,"Common Name"
```

Data rows begin on line 13.

**Do not hardcode line 12.** Scan for the header by looking for a line containing `"Scientific Name"`, then slice from there. This is defensive against minor formatting changes between annual releases.

```javascript
let headerIndex = -1;
for (let i = 0; i < lines.length; i++) {
  if (lines[i].includes('Scientific Name')) {
    headerIndex = i;
    break;
  }
}
const dataLines = lines.slice(headerIndex + 1).filter(line => line.trim() !== '');
```

### Column Layout

| Index | Header | Values | Notes |
|-------|--------|--------|-------|
| 0 | Scientific Name | Mixed-case binomial | May include annotations — see below |
| 1 | Family | Mixed-case | e.g., `Sapindaceae` |
| 2 | Acronym | Uppercase 6-char code | e.g., `ACERUB` |
| 3 | Native? | `native` or `non-native` | Lowercase text, not a code |
| 4 | C | Integer 0–10 | Coefficient of Conservatism |
| 5 | W | Integer -5 to +5 | Coefficient of Wetness |
| 6 | Physiognomy | Mixed string | e.g., `tree`, `forb`, `shrub` — see physiognomy notes |
| 7 | Duration | `annual`, `biennial`, `perennial` | Lowercase |
| 8 | Common Name | String, may have semicolons | e.g., `"red maple"` or `"okra; gumbo"` |

**Real data rows** (from the file):
```
"Acer rubrum",Sapindaceae,ACERUB,native,1,0,tree,perennial,"red maple"
"Abelmoschus esculentus",Malvaceae,ABEESC,non-native,0,5,forb,annual,"okra; gumbo"
"Abies balsamea",Pinaceae,ABIBAL,native,3,0,tree,perennial,"balsam fir"
```

### Scientific Name Annotations in the CSV

Some CSV entries contain semicolons in the scientific name field:

```
"Acer nigrum; a. saccharum"
```

This is a taxonomic annotation, not a delimiter between species. The text after `;` is a reference to a closely related species under Michigan Flora's treatment. When building a lookup index, split at `;` and use only the first token:

```javascript
function parseScientificName(scientificName) {
  const cleanName = scientificName.split(';')[0].trim();
  const parts = cleanName.split(' ');
  return { genus: parts[0] || '', species: parts[1] || '' };
}
```

### Native Status in the CSV

The CSV `Native?` field contains the strings `"native"` or `"non-native"` (lowercase, full words). This differs from the API which uses single-letter codes (`"N"` / `"A"`). Convert to a boolean with a lowercase comparison:

```javascript
isNative: nativeStatus?.toLowerCase() === 'native'
```

### Physiognomy Values

CSV physiognomy values are plain words like `tree`, `forb`, `shrub`, `grass`, `vine`, `fern`. These differ from the API's physiognomy field (see API section), which prefixes native status: `"Nt Tree"`, `"Ad Shrub"`. The CSV and API physiognomy fields are not identical.

### Memory Loading Strategy

The dataset is loaded once at startup and held in two in-memory structures:

1. **Array** (`cachedDataset`): all 2,873 records in order
2. **Map** (`datasetByName`): two entries per record for flexible lookup —
   - Parsed key: `"acer rubrum"` (lowercased genus + species from annotation-stripped name)
   - Full name key: `"acer rubrum"` (the complete `scientificName` field, lowercased)

Both keys are often identical, but diverge for annotated names (the full name key would be `"acer nigrum; a. saccharum"` — an unlikely lookup target, while the parsed key `"acer nigrum"` is the practical one).

Lookup is always case-insensitive:

```javascript
const key = `${genus} ${species}`.toLowerCase().trim();
return datasetByName.get(key) || null;
```

## Interface 2: The REST API

### Base URL

```
https://michiganflora.net/api/v1.0/
```

### Authentication

None required. Include a `User-Agent` header identifying your application:
```
User-Agent: BotanicalDataAggregator/1.0
```

No rate limiting has been observed in practice. Cache all responses — see the caching section.

### Workflow: The plant_id Is the Key

All secondary endpoints require a `plant_id` integer, which is Michigan Flora's primary key for a species record. You cannot construct or guess this ID from the species name — it must be obtained by first calling `flora_search_sp`.

The standard workflow is:

```
flora_search_sp?scientific_name=Genus species
  → extracts plant_id from result[0]
  → then call spec_text, synonyms, locs_sp in parallel using plant_id
```

### Endpoint 1: `flora_search_sp` — Species Search

```
GET /api/v1.0/flora_search_sp?scientific_name={Genus species}
```

Returns an **array** (not an object) of matching species. For a binomial lookup, there will typically be exactly one result. Take `result[0]`.

**Response:**
```json
[
  {
    "plant_id": 2654,
    "scientific_name": "Acer rubrum",
    "c": "1",
    "st": "NULL",
    "w": 0,
    "wet": "FAC",
    "phys": "Nt Tree",
    "na": "N",
    "family_name": "Sapindaceae",
    "author": "L.",
    "acronym": "ACERUB",
    "common_name": ["RED MAPLE"]
  }
]
```

**Field details:**

| Field | Type | Description |
|-------|------|-------------|
| `plant_id` | integer | Primary key — required for all other endpoints |
| `scientific_name` | string | Binomial name in mixed-case (see casing note below) |
| `c` | string | Coefficient of Conservatism as a **string**, not integer. `"*"` for non-native species |
| `st` | string | State status: `"T"` (Threatened), `"E"` (Endangered), `"SC"` (Special Concern), or `"NULL"` as a literal string |
| `w` | number | Wetness coefficient as a **number** (not string), range -5 to +5 |
| `wet` | string | Wetness category: `OBL`, `FACW`, `FAC`, `FACU`, `UPL` |
| `phys` | string | Physiognomy with native prefix: `"Nt Tree"`, `"Ad Shrub"`, `"Nt P-Forb"`, etc. |
| `na` | string | `"N"` = Native, `"A"` = Adventive (non-native/introduced) |
| `family_name` | string | Taxonomic family |
| `author` | string | Taxonomic authority (e.g., `"L."` for Linnaeus) |
| `acronym` | string | 6-character Michigan Flora code |
| `common_name` | string array | Common names in **UPPERCASE**; may have multiple entries |

**Type inconsistency — `c` vs `w`**: The `c` field is returned as a string (`"1"`), while `w` is returned as a number (`0`). Do not assume they share a type.

**`st` returns `"NULL"` as a string**: The state status field returns the four characters `NULL` when there is no status — not a JavaScript `null` value, not an empty string. Check `apiData.st !== 'NULL'` before treating it as meaningful data.

**`c` is `"*"` for non-native species**: When a species is non-native, the `c` field is the string `"*"` rather than a numeric string. The CSV uses integers directly; the API uses strings. Convert carefully:

```javascript
// Correct: preserve "*" as-is, convert numerics
const coefficientC = apiData.c === '*' ? '*' : parseInt(apiData.c, 10);

// Wrong: parseInt("*") = NaN
```

**`phys` format in the API vs CSV**: The API physiognomy field prefixes native/adventive status:
- `"Nt Tree"` = Native Tree
- `"Ad Shrub"` = Adventive Shrub
- `"Nt P-Forb"` = Native Perennial Forb

The CSV's `Physiognomy` column uses only the type (`tree`, `forb`, etc.) without the prefix. They describe the same species but are not directly interchangeable strings.

### Scientific Name Casing — API vs Documentation

Early documentation (including `docs/michigan-flora-api.md`) shows examples with UPPERCASE scientific names (`"PHILADELPHUS INODORUS"`). Actual cached API responses show mixed-case names consistent with the CSV (`"Acer rubrum"`). Do not write code that assumes uppercase scientific names from the API. Use case-insensitive comparisons for any name matching.

### Endpoint 2: `spec_text` — Species Description

```
GET /api/v1.0/spec_text?id={plant_id}
```

Returns a prose description of the species — habitat, distribution, identification notes.

**Response:**
```json
{
  "text": "<p>In almost all sorts of forests and savannas, commonly in moist forests and swamps...</p>\n\n<p>This is a well-named tree, for the flowers, young fruit, and autumn foliage are often a brilliant red...</p>"
}
```

**Key details:**
- The `text` field is **HTML**, not plain text — it uses `<p>` tags and may include `<i>`, `<b>`, etc.
- Some species have no description — the field may be null or an empty string. Always guard:
  ```javascript
  if (textResult?.text) { miFloraDescription = stripHtml(textResult.text); }
  ```
- HTML stripping should handle common entities: `&nbsp;`, `&quot;`, `&amp;`, `&lt;`, `&gt;`, `&plusmn;`

---

### Endpoint 3: `synonyms` — Botanical Synonyms

```
GET /api/v1.0/synonyms?id={plant_id}
```

**Response when synonyms exist:**
```json
{
  "synonyms": [
    {
      "syn_species": "saccharum",
      "syn_genus": "Acer",
      "syn_notes": "split from",
      "synonym_id": 390
    }
  ]
}
```

**Response when no synonyms:**
```json
{
  "message": "No synonyms found"
}
```

The "no synonyms" response is a completely different object shape — it has a `message` key instead of a `synonyms` key. Extract safely:

```javascript
const synonyms = result?.synonyms || [];
```

The `syn_notes` field may be the string `"NULL"` (same literal-string-NULL pattern as `st`). The `syn_genus` + `syn_species` fields together reconstruct the synonym binomial.

**Real examples**:
- *Acer saccharum* has synonym `Acer saccharum` from itself (split from), `syn_notes: "split from"`
- *Eurybia macrophylla* has synonym `Aster macrophyllus`, `syn_notes: "NULL"`

---

### Endpoint 4: `locs_sp` — County Occurrences

```
GET /api/v1.0/locs_sp?id={plant_id}
```

Returns the list of Michigan counties where the species has documented occurrences, drawn from herbarium records.

**Response:**
```json
{
  "locations": ["Alcona", "Alger", "Allegan", "Alpena", ...]
}
```

The response is a flat array of county name strings. County names are in mixed case with standard spelling. Widespread species like *Acer rubrum* can appear in 78 of Michigan's 83 counties. Rare species may appear in only 1–3.

This data is most useful for:
- Confirming SE Michigan presence (check for Washtenaw, Wayne, Oakland, Monroe, Livingston, etc.)
- Understanding range (statewide vs. northern Michigan only vs. southern Michigan only)
- Identifying locally rare species in the SE Michigan context

---

### Endpoint 5: `pimage_info` / `allimage_info` — Images

```
GET /api/v1.0/pimage_info?id={plant_id}
GET /api/v1.0/allimage_info?id={plant_id}
```

Returns image metadata but not image data directly. The `image_id` and `image_name` fields are present but no image URL is returned by the API. Image URL construction from these fields was not explored; actual image hosting is on the Michigan Flora server. These endpoints are cached but not used in the current synthesis pipeline.

---

### Endpoint 6: Record URL Construction

The URL to a species page on the Michigan Flora website follows a fixed pattern:

```
https://michiganflora.net/record/{plant_id}
```

Example: `https://michiganflora.net/record/2654` → *Acer rubrum*

This URL is used as the Michigan Flora external reference link added to the pipeline's URL collection. The page itself is a Vue.js SPA and requires JavaScript to render; the URL is suitable for human browsing but cannot be scraped.

### Additional Endpoints (Available but Not Used in Pipeline)

| Endpoint | Parameters | Returns |
|----------|-----------|---------|
| `genus_list` | `family` (optional) | Array of all genera (optionally in a family) |
| `family_list` | none | Array of all plant families |
| `genus_text?id={genus}` | genus name string | HTML text with genus description and dichotomous key |
| `family_text?id={family}` | family name string | HTML text with family description |

Note: `genus_text` returns Vue.js router-link components embedded in the HTML for cross-references to species pages. These are Vue-specific markup and should be stripped or ignored in text processing contexts.

## Data Source Selection Strategy

The synthesis module uses both interfaces together, with a defined resolution order:

**For C (Coefficient of Conservatism):**
- Primary: CSV `C` column
- Fallback: API `c` field (only if species not found in CSV and not `"*"`)

**For W (Coefficient of Wetness):**
- Primary: CSV `W` column
- Fallback: API `w` field (only if species not found in CSV)

**For Physiognomy and Duration:**
- Primary: CSV (only source for Duration; API doesn't have it)
- Fallback for Physiognomy: API `phys` field (only if not in CSV)

**For Native Status:**
- Primary: API `na` field (`"N"` → `"Native"`, `"A"` → `"Non-native"`)
- The CSV `Native?` field also provides this but the API is preferred since it also gates other API data

**For Description Text:**
- Exclusive: API `spec_text` endpoint

**For County Occurrences:**
- Exclusive: API `locs_sp` endpoint

**For Record URL:**
- Derived from API `plant_id`: `https://michiganflora.net/record/{plant_id}`

If the species is found in the API but not the CSV (uncommon but possible for recently added or obscure species), the API's ecological fields are used as a fallback. If the species is found in the CSV but not the API (can happen for synonyms/variants), only CSV fields are available and the description/URL/county data will be absent.

## Caching

### API Cache

Four separate JSON files per species in `cache/MichiganFlora/API/`:

| Suffix | Endpoint | Contents |
|--------|----------|---------|
| `_flora.json` | `flora_search_sp` | Full search result including `plant_id`, ecological fields |
| `_text.json` | `spec_text` | Species description (raw HTML) |
| `_synonyms.json` | `synonyms` | Synonym array (or empty) |
| `_locations.json` | `locs_sp` | County array |

**Cache file naming**: `{Genus}_{species}_{suffix}.json` — e.g., `Acer_rubrum_flora.json`, `Acer_rubrum_text.json`

Genus normalization: first letter uppercase, rest lowercase. Species: all lowercase. Subspecies or varieties in the species epithet are slugified (spaces → hyphens, special characters → underscores; the `×` hybrid marker is preserved).

**Not-found results are cached** (`{ found: false, data: null }`) to prevent repeat failed API calls.

All files include a `fetchedAt` ISO timestamp to track when the data was obtained.

### Composite SpeciesData JSON

After a successful API lookup, the synthesis module writes a comprehensive summary JSON per species:

```
cache/MichiganFlora/SpeciesData/{Genus}_{species}_miflora.json
```

**Real example** for *Acer rubrum*:
```json
{
  "_meta": {
    "genus": "Acer",
    "species": "rubrum",
    "source": "Michigan Flora",
    "recordUrl": "https://michiganflora.net/record/2654",
    "plantId": 2654,
    "fetchedAt": "2025-12-26T18:27:30.409Z"
  },
  "basicData": {
    "scientificName": "Acer rubrum",
    "familyName": "Sapindaceae",
    "author": "L.",
    "acronym": "ACERUB",
    "commonNames": ["RED MAPLE"]
  },
  "ecologicalData": {
    "coefficientC": "1",
    "wetnessCoefficient": 0,
    "wetnessIndicator": "FAC",
    "physiognomy": "Nt Tree",
    "nativeStatus": "N",
    "stateStatus": "NULL"
  },
  "description": "In almost all sorts of forests and savannas...",
  "descriptionHtml": "<p>In almost all sorts of forests and savannas...</p>",
  "synonyms": [],
  "counties": ["Alcona", "Alger", ... "Wayne"]
}
```

This composite JSON is the artifact used by 3-tier LLM synthesis as the Tier 2 secondary source for Michigan Flora data. It stores both the plain-text description (with HTML stripped) and the original HTML for different downstream consumers.

### CSV Cache

The CSV is loaded once per process into a module-level singleton (`cachedDataset` array + `datasetByName` Map). Subsequent calls to `loadDataset()` return the cached result immediately. There is no TTL — the data lives for the lifetime of the process. To force a reload during testing, call `clearCache()` to reset both singletons.

### Google Drive Upload

As a side effect of the synthesis module run, the composite SpeciesData JSON is uploaded to Google Drive under a `MichiganFloraData` subfolder. This upload is explicitly non-fatal — if it fails (network error, auth issue, quota), a warning is logged and the pipeline continues. The local cache file is always written first, independently.

## Known Behaviors and Edge Cases

### The "NULL" String

Both `st` (state status) and `syn_notes` (synonym notes) return the literal four-character string `"NULL"` when there is no value — not JavaScript `null`, not an empty string, not `undefined`. Always check `value !== 'NULL'` before treating these fields as meaningful:

```javascript
// Check state status
if (apiData.st && apiData.st !== 'NULL') {
  // species has a state conservation status
}
```

### `c` Is a String, `w` Is a Number

The coefficient of conservatism (`c`) and wetness coefficient (`w`) are typed differently in the API response even though both are numeric concepts:
- `c`: always a string — `"1"`, `"7"`, `"*"`
- `w`: always a number — `0`, `-3`, `5`

The CSV parses both as integers via `parseInt()`, which works for numeric strings but returns `NaN` for `"*"`. The synthesis module uses `String(coefficientC)` when writing to the pipeline to normalize both to strings for consistent output.

### Species Not in CSV But Found in API

A species may be present in the API but absent from the CSV. This can happen for:
- Very recently added species
- Species recorded only in the Upper Peninsula (the CSV may weight southern Michigan)
- Hybrids or subspecies treated differently across the two data sources

When this occurs, the synthesis module falls back to the API for C and W values, but these are typed as strings from the API (`"c": "3"`) whereas CSV values are integers. The fallback path converts: `apiData.c !== '*' ? apiData.c : ''`.

### Species Not in CSV AND Not in API

If both lookups fail, all ecological metric fields are returned as empty strings and no record URL is appended. This is not treated as a pipeline error — the module returns gracefully with empty values for its fields. The dependency on `botanical-name` ensures the species name is valid per GBIF before Michigan Flora is queried, but GBIF and Michigan Flora are independent taxonomic databases and may disagree on what constitutes a valid species.

### Common Names Are UPPERCASE in API, Mixed Case in CSV

The API `common_name` array returns all uppercase names (`["RED MAPLE"]`). The CSV `Common Name` field uses sentence case (`"red maple"`). If displaying common names from either source, normalize to a consistent case before display.

### County Data Is Statewide, Not Filtered

The `locs_sp` endpoint returns all documented Michigan counties — both Upper Peninsula and Lower Peninsula. If your use case is SE Michigan specifically, filter the county list client-side (check for Washtenaw, Wayne, Oakland, Monroe, Livingston, Jackson, Lenawee). Widespread species will show 70–80+ counties.

### Parallel API Calls After Getting plant_id

Once `plant_id` is obtained from `flora_search_sp`, the three secondary calls (`spec_text`, `synonyms`, `locs_sp`) have no dependency on each other and can execute in parallel:

```javascript
const [textResult, synonymsResult, locationsResult] = await Promise.all([
  getSpeciesText(plantId, genus, species),
  getSynonyms(plantId, genus, species),
  getLocations(plantId, genus, species)
]);
```

This is the pattern used in production and reduces total API time to the duration of the slowest call.

## Ecological Metric Reference

### Coefficient of Conservatism (C)

A measure of a species' fidelity to high-quality natural areas. Assigned by botanists based on species behavior in Michigan:

| C Value | Interpretation |
|---------|---------------|
| 0 | Found in any habitat, highly tolerant of disturbance |
| 1–3 | Low fidelity; occurs in disturbed and natural habitats |
| 4–6 | Moderate fidelity; prefers natural habitats |
| 7–9 | High fidelity; rarely in disturbed areas |
| 10 | Found only in the highest-quality natural areas |
| `*` | Non-native species; C value not applicable |

Native mean C across all Michigan native species: **6.5** (from the CSV metadata).

### Coefficient of Wetness (W)

A numeric measure of wetland affinity:

| W Value | Indicator Code | Interpretation |
|---------|---------------|----------------|
| -5 | OBL | Obligate wetland (almost always in wetlands) |
| -4 | — | — |
| -3 | FACW | Facultative wetland (usually in wetlands) |
| 0 | FAC | Facultative (equally in wet or dry) |
| 3 | FACU | Facultative upland (usually in uplands) |
| 5 | UPL | Obligate upland (almost never in wetlands) |

W codes follow standard US Fish and Wildlife Service Wetland Indicator categories.

### Native Status (Michigan-specific)

- **Native** (`na: "N"`): Species indigenous to Michigan prior to European settlement
- **Adventive** (`na: "A"`): Non-native, introduced species — may be naturalized but not native

Michigan Flora uses "Adventive" rather than "introduced" or "exotic." The pipeline converts this to `"Non-native"` for clarity in output.

### Physiognomy

The API `phys` field uses a two-part code: native/adventive prefix + growth form. The CSV uses only the growth form.

Common values (API format):
- `Nt Tree`, `Ad Tree` — tree
- `Nt Shrub`, `Ad Shrub` — shrub
- `Nt P-Forb` — Native Perennial Forb
- `Nt A-Forb` — Native Annual Forb
- `Nt Grass` — native grass
- `Nt Sedge` — native sedge (Carex and allies)
- `Nt Fern` — native fern

## What a Complete Connector Needs

1. **CSV loader**: scan for header line (not hardcoded offset), parse with quoted-field awareness, build two in-memory maps (genus+species key, full name key), cache across calls
2. **Name normalization**: strip annotations at `;`, lowercase everything for lookup, handle the `*` C-value case without parseInt errors
3. **API client**: fetch `flora_search_sp` → extract `plant_id` → run `spec_text`, `synonyms`, `locs_sp` in parallel
4. **Type awareness**: `c` is a string, `w` is a number; `st` and `syn_notes` return `"NULL"` as a literal string
5. **HTML stripping**: `spec_text` returns HTML; strip tags and decode entities for plain-text output
6. **Four separate API cache files** per species; plus a composite SpeciesData JSON that merges everything
7. **Resolution order**: CSV for C/W/physiognomy/duration, API for native status/description/URL/counties, API as fallback for C/W/physiognomy if not in CSV
