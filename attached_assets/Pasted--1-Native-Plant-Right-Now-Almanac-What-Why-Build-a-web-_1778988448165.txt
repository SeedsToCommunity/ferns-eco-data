#1 - Native Plant Right-Now Almanac
What & Why
Build a web app called "Native Plant Right-Now Almanac" that helps naturalists and curious locals discover what native plants are doing right now in Washtenaw County and surrounding region. The user picks a place, sees a ranked list of native plant species with active phenology from iNaturalist, selects a species to navigate to a full detail page, and experiences a carefully timed three-stage progressive reveal — instant species info, then a visual BONAP distribution map, then a streaming Claude-generated natural history story — so there is always something engaging on screen.

The Ecological Commons API (data.ecologicalcommons.org) is the preferred data source for all iNaturalist calls it can serve. Where EC does not wrap a needed endpoint, fall back to direct iNaturalist API calls. This is a development-time decision documented in the code.

Done looks like
App is named "Native Plant Right-Now Almanac" throughout
A place selector dropdown shows ~12 curated locations: Washtenaw County + 7 surrounding Michigan counties (Monroe, Wayne, Oakland, Livingston, Ingham, Jackson, Lenawee) + Matthaei Botanical Gardens, County Farm Park (Ann Arbor), Nichols Arboretum, Parker Mill County Park, Bird Hills Nature Area
After selecting a place, a ranked list of native plant species appears, sorted by active phenology observation count in a ±14-day window around today's calendar date across all years
Only interesting phenology shown — filters out "No Flowers or Fruits" (term_value_id=21) and "No Live Leaves" (term_value_id=40)
Top 25 species shown by default with a "show all" option
Each species card shows: common name, scientific name, observation count, thumbnail photo, and active phenology annotation pills
Clicking a species immediately navigates to the detail page and fires three things simultaneously:
A GET /api/phenology/species/:taxonName/details call for enriched iNaturalist data
A GET /api/bonap-map/:genus/:species call for the distribution map image
A POST /api/story streaming call to Claude
The detail page delivers a three-stage progressive reveal (see below)
Story length (Short / Medium / Long) and breadth (Focused / Broad) controls are visible on the detail page; Medium + Broad are pre-selected defaults
Detail page — three-stage progressive reveal
This is the core UX of the app. Each stage gives the user something genuinely interesting while the next, slower stage loads.

Stage 1 — Instant (zero wait) Data already in hand from the list card. Renders immediately on navigation:

Hero photo (full-width or prominent)
Common name + scientific name
Active phenology annotation badges (e.g. "Flowering", "Flower Buds") in a clear visual treatment
Observation count and place name
iNaturalist link
Stage 2 — Fast (1–3 seconds, visual) BONAP call returns an image URL pointing to a county-level distribution map PNG showing exactly where this plant exists across North America. Renders as a visual map section below Stage 1. This gives the user something compelling to look at — a continental picture of this plant's range — while Claude is still composing. The map section shows a skeleton/loading state until the image URL arrives, then fades in.

Stage 3 — Streaming (starts in parallel, prose appears progressively) Claude story streams in word by word into the third section. By the time the first sentences arrive, Stages 1 and 2 are already visible — so the streaming text feels like a natural arrival, not a blank wait. A subtle "composing..." indicator gives way to the streaming prose as soon as the first tokens arrive. The full iNaturalist detail data (Wikipedia summary, phenology histogram, recent observations) also loads into this section as it arrives from the enrichment call.

Layout: on desktop, Stage 2 (map) and Stage 3 (story) can sit side by side below Stage 1, or stacked — the DESIGN subagent decides. The timing choreography matters more than the exact layout.

Out of scope
Any Ecological Commons endpoints other than /inat/* and /bonap/map
User accounts or saved preferences
Animal species (plants only)
Custom date range picker
Offline support
EC vs. Direct iNaturalist — development-time routing decisions
Data needed	Source	Endpoint
Ranked species list by phenology count	Direct iNaturalist	api.inaturalist.org/v1/observations/species_counts (EC does not wrap this)
Place name → place_id lookup	EC	data.ecologicalcommons.org/api/inat/place?name=
Species photos, Wikipedia summary, monthly counts, phenology breakdown	EC	data.ecologicalcommons.org/api/inat/species?name=&place_id=
Phenology histogram (by month, filtered to interesting annotations)	EC	data.ecologicalcommons.org/api/inat/histogram (passthrough)
Popular field values	EC	data.ecologicalcommons.org/api/inat/field-values?taxon_name=&place_id=
Recent observation records	EC	data.ecologicalcommons.org/api/inat/observation-summary?taxon_name=&place_id=
BONAP county distribution map	EC	data.ecologicalcommons.org/api/bonap/map?genus=&species=&map_type=county_species
Steps
Set up Replit Anthropic AI Integration — Use Replit's managed Claude integration (no user API key needed; billed to Replit credits; auto-provisioned). In build mode: call setupReplitAIIntegrations({ providerSlug: "anthropic", providerUrlEnvVarName: "AI_INTEGRATIONS_ANTHROPIC_BASE_URL", providerApiKeyEnvVarName: "AI_INTEGRATIONS_ANTHROPIC_API_KEY" }). Copy templates from .local/skills/ai-integrations-anthropic/templates/lib/* into lib/. Add @workspace/integrations-anthropic-ai dependency to API server package.json. Run pnpm install --no-frozen-lockfile. Add TS project references to root tsconfig.json and artifacts/api-server/tsconfig.json. Export conversations/messages schema from lib/db/src/schema/index.ts. Run pnpm --filter @workspace/db run push. Follow .local/skills/ai-integrations-anthropic/SKILL.md exactly.

Backend: resolve and hardcode place IDs — Before writing any route code, call GET https://data.ecologicalcommons.org/api/inat/place?name={placeName} for each of the 7 surrounding counties to resolve their iNaturalist place IDs. Confirmed IDs: Washtenaw County=2649, Matthaei Botanical Gardens=198456, County Farm Park (Ann Arbor)=141499, Parker Mill County Park=202861, Bird Hills Nature Area=160435. Nichols Arboretum: resolve via EC. Create a shared place config object in the API server used by both the places route and the phenology routes.

Backend: places route — GET /api/places returns the hardcoded curated place list (name, iNaturalist place_id, short description). No external call at runtime.

Backend: phenology species list route — GET /api/phenology/species?place_id=&limit=25 calls direct iNaturalist at api.inaturalist.org/v1/observations/species_counts with: place_id, taxon_id=47126 (Plants), term_id=12 (Flowers and Fruits), without_term_value_id=21, native=true, verifiable=true, and a ±14-day date window. Compute d1/d2 as today's month+day ±14 days using fixed reference year 2024 (iNaturalist treats this as a calendar-day filter across all observation years). Handle year-boundary wrap. Return ranked array with taxon id, scientific name, common name, observation count, default photo url, and active annotation value labels.

Backend: species detail enrichment route — GET /api/phenology/species/:taxonName/details?place_id= calls these EC endpoints in parallel:

GET /api/inat/species?name={name}&place_id={id} — photos, Wikipedia summary, monthly counts, phenology breakdown, native status
GET /api/inat/histogram?taxon_name={name}&place_id={id}&term_id=12&without_term_value_id=21 — phenology histogram for charting
GET /api/inat/field-values?taxon_name={name}&place_id={id} — popular field values
GET /api/inat/observation-summary?taxon_name={name}&place_id={id}&per_page=6 — recent observations with photos Assemble into one response, gracefully handling partial failures.
Backend: BONAP map route — GET /api/bonap-map/:genus/:species calls EC at data.ecologicalcommons.org/api/bonap/map?genus={genus}&species={species}&map_type=county_species. Parse the genus and species from the scientific name (split on first space). Returns { mapUrl, colorKey, colorKeyImageUrl, attribution } so the frontend can render the map image and display attribution.

Backend: Claude story generation route — POST /api/story accepts { taxonName, commonName, placeName, placeId, speciesDetails, storyLength, storyBreadth }. Build a Claude prompt including: common and scientific name, the specific named place, today's full date and season, active phenology annotations (what the plant is doing right now), monthly observation histogram narrative, Wikipedia summary, notable field values, and recent observation context. Instruct claude-sonnet-4-6 to write a vivid, grounded natural history story about this plant at this specific place in this season. Stream via SSE using anthropic.messages.stream() — send tokens as they arrive, do not buffer. Respect storyLength (short/medium/long → ~150/350/600 words) and storyBreadth (focused=this phenology moment only vs broad=full life cycle, ecology, seasonal lore). Set max_tokens: 8192. See .local/skills/ai-integrations-anthropic/SKILL.md for SSE streaming pattern.

OpenAPI spec + codegen — Update lib/api-spec/openapi.yaml with: GET /api/places, GET /api/phenology/species, GET /api/phenology/species/{taxonName}/details, GET /api/bonap-map/{genus}/{species}, POST /api/story, plus Anthropic paths from .local/skills/ai-integrations-anthropic/references/openapi.md. Run pnpm --filter @workspace/api-spec run codegen.

Frontend: React Vite app — Create a react-vite artifact at / (root). After codegen, launch the DESIGN subagent with the creative brief. The app has two views: home (place selector + species list) and species detail page. The app is mobile-first. Design for a phone screen as the primary target — touch-friendly tap targets, full-width cards, bottom-friendly navigation, thumb-reachable controls. Desktop layout is a graceful enhancement, not the baseline.

Frontend detail page — three-stage progressive reveal — The DESIGN subagent must implement this timing carefully. On species click, navigate immediately to the detail page and fire all three API calls simultaneously. Render the three stages as they resolve:

Stage 1 (instant): Hero photo, name, phenology badges, count, place — all from data already held in the list state. No loading state needed here.
Stage 2 (fast): BONAP map section shows a skeleton until the map URL arrives, then fades in the county distribution image with BONAP attribution. This visual loads well before Claude's first tokens. On mobile, the map renders full-width.
Stage 3 (streaming): Story section shows a subtle "composing..." state that transitions to streaming prose as soon as first Claude tokens arrive. The full enrichment data (Wikipedia, histogram chart, observation records) loads into the page as the detail fetch completes — staggered reveals are appropriate. Story controls (length/breadth) are visible but greyed out while streaming. On mobile, all sections stack vertically in this order: Stage 1 → Stage 2 → Stage 3.
